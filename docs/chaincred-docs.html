ChainCred
Documentation

ChainCred is an onchain expertise reputation layer for Ethereum and EVM networks. It indexes wallet activity, computes sybil-resistant expertise scores from 0 to 1000, and makes results available through a REST API, web explorer, and EAS attestations.

Scores are based on what wallets do — not what they hold.

How Scoring Works

Every wallet receives a score from 0 to 1000. The score is computed from five weighted categories, each producing a raw score (0–1000) from multiple onchain signals. The raw weighted sum is then multiplied by a sybil confidence multiplier (0.0–1.0).

rawScore = builder.weighted + governance.weighted + temporal.weighted + diversity.weighted + complexity.weighted
totalScore = round(rawScore × sybilMultiplier)

Sybil Detection

Every wallet is analyzed for bot-like behavior using 10 heuristics. Each detected flag reduces the confidence multiplier multiplicatively. Wallets are never banned — just discounted. The confidence score and all detected flags are always visible.

confidence = 1.0
for each detected flag:
  confidence ×= (1 − penalty)

totalScore = round(rawScore × confidence)
1. Temporal Clustering −40%

Fires when avg >20 txs/day AND wallet age <90 days. 500 txs over 3 years is normal; 500 in 2 weeks is suspicious.

2. Action Repetition −30%

Fires when >100 txs but zero recognized protocols on a single chain. Wallets using at least one known protocol are exempt.

3. Funding Graph Clustering −50%

Fires when >10 unique recipients but <3 protocols. Detects sybil cluster coordinators distributing funds.

4. Cross-Chain Mirroring −60%

Fires when 3+ chains share an identical protocol set. Automated sybil farms replicate the same interactions across chains.

5. CEX Withdrawal Freshness up to −30%

Graduated penalty for wallets <30 days old. Penalty = 30% × (1 − ageDays/30). Zero penalty after 30 days.

6. Zero Failure Rate −20%

Fires when >200 txs and zero failures. Real users inevitably encounter reverts. Perfect success at volume suggests automated pre-simulation.

7. Perfect Gas Patterns −15%

Fires when >50 txs and <5% distinct gas prices. Bots hardcode gas; humans encounter varying network conditions.

8. MEV Bot Activity −15%

Fires when >20 MEV interactions AND >30% of total txs. Tracks 6 known MEV contract addresses. Both conditions required (conservative).

9. Funding Source Cluster −50%

Uses Etherscan to find the first incoming tx funder. If that funder sent to >10 unique wallets, this wallet is likely part of a sybil cluster. Requires ETHERSCAN_API_KEY.

10. CEX Fresh Wallet up to −30%

Graduated penalty for wallets funded by a known CEX hot wallet (Binance, Coinbase, Kraken, OKX, Gemini) and <90 days old. Requires ETHERSCAN_API_KEY.

Sybil Appeal System

Wallets flagged by sybil detection can submit appeals for manual review.

  • Submit: POST /v1/appeals with { address, reason }
  • Check: GET /v1/appeals/:address returns the latest appeal status
  • Score protection: While an appeal is pending, the sybil multiplier is floored at 0.5 — the score is reduced but not zeroed

Appeals are stored in PostgreSQL with status tracking (pending/approved/rejected).

Exclusion Registry

When CHAINCRED_REGISTRY_ADDRESS is set, the score service checks the onchain ChainCredRegistry contract before scoring. Excluded addresses (protocol-owned wallets, known bots) always receive a score of 0.

The check is fail-open — if the RPC is unreachable, scoring proceeds normally. Admins manage the registry via setExcluded(address, bool) on the contract.

WebSocket & Webhooks

WebSocket Streaming

Connect to ws://localhost:3001/v1/ws/stream/:address for real-time score updates. The server sends the current score immediately on connect, then pushes refreshed scores every 60 seconds.

Messages: { type: "score", data: WalletScore }. Supports ping/pong keepalive.

Webhooks

Register persistent webhooks for score update notifications. Webhooks survive API restarts (stored in PostgreSQL).

  • Payload: { event: "score.updated", address, score, timestamp }
  • HMAC-SHA256: If a secret is provided at registration, payloads are signed via X-ChainCred-Signature header
  • Delivery: Best-effort, fired alongside WebSocket pushes

Smart Contracts

Solidity 0.8.24, built with Foundry. Three contracts handle onchain verification and configuration.

ChainCredRegistry

Central configuration contract. Inherits Ownable, Pausable.

  • Stores EAS schema UIDs for score and badge attestations
  • Manages admin roles (setAdmin)
  • Maintains exclusion list for protocol-owned wallets and bots (setExcluded)
  • Pausable by admins, unpausable by owner only

ScoreMerkleRoot

Publishes weekly Merkle roots for trustless score verification. Inherits Ownable.

  • updateMerkleRoot(bytes32) — increments epoch, sets timestamp
  • verifyScore(address, uint256, bytes32[]) — view function, double-hash leaf encoding
  • Leaf: keccak256(keccak256(abi.encode(address, score))) (OpenZeppelin standard)
  • Reverts if score > 1000 (ScoreOutOfRange)

BadgeResolver

EAS SchemaResolver for badge attestations. Inherits Ownable.

  • Validates attestations come from the authorized attestor
  • Registers 7 badge type hashes: builder, governor, explorer, og, multichain, trusted, power-user
  • onAttest / onRevoke callbacks from EAS

Merkle Proofs & IPFS

Merkle Tree

A weekly Merkle tree is generated from all scored wallets. Proofs are stored in PostgreSQL and served via /v1/proof/:address. The root is published onchain via ScoreMerkleRoot.updateMerkleRoot(), enabling trustless verification without trusting the API.

IPFS Score Breakdowns

When PINATA_API_KEY and PINATA_API_SECRET are set, score breakdowns are uploaded to IPFS via Pinata.

  • Score route: Uploads breakdown after scoring, returns breakdownCID
  • Merkle generation: Batch-uploads breakdowns, stores CID in merkle_proofs.breakdown_cid
  • Content-hash cache: Identical breakdowns reuse the same CID (24h TTL)
  • Fail-open: If Pinata is unreachable, scoring proceeds without CID

CIDs resolve via any IPFS gateway: https://gateway.pinata.cloud/ipfs/{CID}

Main Indexer

Syncs all transactions from block 0 across all chains in parallel. Pages of 2,000 blocks, polls every 30 seconds. Persists cursor per chain in indexer_cursors table.

Single-Wallet Indexer

Fast CLI tool for indexing a specific wallet's transactions:

bun run index-wallet 0x887b86B6B6957F7bbeA88B8CEfD392f39236A88C

Scans all 6 chains from block 0 in parallel with wider pages (10,000 blocks). Uses the same enrichment pipeline as the full indexer. Typical runtime: seconds to minutes depending on wallet activity.

Enrichment Pipeline

Every transaction is enriched with:

  • Protocol lookup: Maps tx.to to one of 20 known protocols + category
  • Governance classification: 5 subtypes (vote, propose, delegate, queue, execute)
  • CREATE2 detection: 4 known factory addresses
  • Selector detection: Permits, flashloans, ERC-4337, Safe execTransaction, reasoned votes
  • Smart wallet detection: Calls to ERC-4337 EntryPoint v0.6/v0.7
  • MEV detection: 6 known MEV contract addresses
  • Early adoption: Protocol used within 6 months of launch
  • Vote support parsing: Extracts against/abstain from castVote calldata

ENS Resolution

Reverse ENS resolution via getEnsName() on Ethereum mainnet. Names are cached in-memory for 1 hour. Fail-open.

ENS names appear in: the score API response (ensName field), the SVG score card, and the frontend score page.
